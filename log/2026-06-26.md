## Dev-only: bridge a local Anthropic key into the client (PR #8)

`vite.config.ts` now reads `.env.local`'s `ANTHROPIC_API_KEY` via `loadEnv` and
injects it into the client through a `__DEV_ANTHROPIC_KEY__` define — but **only**
for `vite dev` (`command === 'serve'`); every build emits `""`. `client.ts` gained
`resolveApiKey()` (browser-stored key wins, else the dev value behind an
`import.meta.env.DEV` guard); `hasKey()` and `callAnthropic` both route through it.
So locally the AI affordances (incl. the dev eval bar) light up from `.env.local`
with no UI key entry, while a hosted build never embeds or uses the key.

Verified: prod build passes **with and without** `.env.local` (the Vercel
checkout case); the key string is absent from `.output`; `tsc`/`eslint`/73 tests
green; and the Vercel **preview deploy succeeded** with the AI surfaces correctly
absent.

**Why:** convenience without a hosting footgun — one env var already used by
`scripts/eval.mjs` now also lights up the in-app dev tooling, gated so it can
never leak into or gate the deployed app.
